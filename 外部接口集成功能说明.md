# 外部接口集成功能说明

## 功能概述

在DataCollectService服务中集成了外部接口调用功能，用于在采集任务下发时检查应用是否为新应用，并根据检查结果自动添加自定义参数。

## 实现的功能

### 1. 外部接口调用服务

- **服务类**: `ExternalApiService` 和 `ExternalApiServiceImpl`
- **控制器**: `ExternalApiController`
- **DTO类**: `AppCheckRequest` 和 `AppCheckResponse`

### 2. 配置

在 `application.yml` 中添加了外部接口配置：

```yaml
# 外部接口配置
external:
  api:
    host: http://localhost:9000
```

### 3. 集成到采集任务下发流程

在 `CollectTaskProcessServiceImpl.getTaskCustomParams()` 方法中集成了 `checkAppIsNew` 方法：

- 收集所有唯一的APP名称
- 为每个APP查询iOS和非iOS两种版本（is_ios为true和false）
- 通过用例编号是否包含 `_ios_` 来判断用例是否为iOS应用
- 通过用例APP字段和is_ios状态匹配查询结果
- 根据匹配的 `is_new` 值添加自定义参数：
  - 如果 `is_new` 为 `true`，添加参数 `key: "isnew", value: "1"`
  - 如果 `is_new` 为 `false`，添加参数 `key: "isnew", value: "0"`
  - 如果没有查询到结果，默认添加参数 `key: "isnew", value: "0"`

### 4. 集成到用例执行结果上报流程

在 `TestCaseExecutionResultController.reportTestCaseResult()` 方法中集成了 `updateProbedStatus` 方法：

- 当用例执行结果上报成功后，检查用例采集状态
- **只有用例采集状态为成功（SUCCESS或COMPLETED）时，才调用 `updateProbedStatus` 接口**
- 根据用例的APP字段获取应用名称
- 调用外部接口更新探测状态，标记用例已采集
- 如果调用失败，不会影响用例执行结果上报的主流程

## 接口规范

### 1. 检查应用是否为新应用

**URL**: `{host}/api/apps/check_is_new`

**方法**: POST

**请求体**:
```json
[
  {
    "app_name": "微信",
    "is_ios": false
  }
]
```

**响应格式**:
```json
{
  "message": "success",
  "data": [
    {
      "app_name": "微信",
      "is_ios": false,
      "is_new": false
    }
  ]
}
```

### 2. 更新探测状态

**URL**: `{host}/api/apps/updata_probed_status`

**方法**: POST

**请求体**:
```json
["微信", "微博"]
```

**响应格式**:
```json
{
  "message": "success",
  "data": {
    "updata_count": 3
  }
}
```

**注意**：接口使用下划线命名（app_name, is_ios, is_new, updata_count），但Java代码中使用驼峰命名（appName, isIos, isNew, updataCount），通过@JsonProperty注解自动转换。

## 使用方式

### 1. 直接调用外部接口

**检查应用是否为新应用**:
```bash
curl -X POST http://localhost:8080/api/external/check-app-is-new \
  -H "Content-Type: application/json" \
  -d '[{"app_name": "微信", "is_ios": false}]'
```

**更新探测状态**:
```bash
curl -X POST http://localhost:8080/api/external/update-probed-status \
  -H "Content-Type: application/json" \
  -d '["微信", "微博"]'
```

### 2. 工作流程示例

假设有以下用例：
- 用例1：编号 `TC001_android_微信登录测试`，APP字段 `微信`
- 用例2：编号 `TC002_ios_微信登录测试`，APP字段 `微信`
- 用例3：编号 `TC003_android_支付宝支付测试`，APP字段 `支付宝`

系统会执行以下步骤：

1. **收集唯一APP**：`微信`、`支付宝`

2. **发送查询请求**：
   ```json
   [
     {"app_name": "微信", "is_ios": true},
     {"app_name": "微信", "is_ios": false},
     {"app_name": "支付宝", "is_ios": true},
     {"app_name": "支付宝", "is_ios": false}
   ]
   ```

3. **接收响应**：
   ```json
   {
     "message": "success",
     "data": [
       {"app_name": "微信", "is_ios": true, "is_new": true},
       {"app_name": "微信", "is_ios": false, "is_new": false},
       {"app_name": "支付宝", "is_ios": true, "is_new": false},
       {"app_name": "支付宝", "is_ios": false, "is_new": true}
     ]
   }
   ```

4. **匹配结果**：
   - 用例1：`微信` + `is_ios: false` → `is_new: false` → 添加参数 `isnew: 0`
   - 用例2：`微信` + `is_ios: true` → `is_new: true` → 添加参数 `isnew: 1`
   - 用例3：`支付宝` + `is_ios: false` → `is_new: true` → 添加参数 `isnew: 1`

### 3. 在采集任务中自动调用

当创建采集任务时，系统会自动：

1. 收集所有唯一的APP名称
2. 为每个APP查询iOS和非iOS两种版本（发送is_ios为true和false的请求）
3. 根据用例编号判断是否为iOS应用（包含 `_ios_` 关键字）
4. 通过用例APP字段和is_ios状态匹配查询结果
5. 将匹配的检查结果作为自定义参数添加到任务中
6. 如果没有查询到结果，使用默认值isnew=0

### 4. 在用例执行结果上报时自动调用

当用例执行结果上报成功时，系统会自动：

1. 保存用例执行结果到数据库
2. **检查用例采集状态是否为成功（SUCCESS或COMPLETED）**
3. 如果状态为成功，获取用例的APP字段信息
4. 调用 `updateProbedStatus` 接口更新探测状态
5. 记录调用结果和更新数量
6. 如果状态不是成功，跳过调用并记录日志
7. 如果调用失败，记录错误日志但不影响主流程

## 日志记录

系统会记录以下关键日志：

- 外部接口调用请求和响应
- 用例的APP信息和is_new状态
- 自定义参数的添加过程
- 用例执行结果上报成功后的状态检查
- 只有成功状态才调用updateProbedStatus接口
- updateProbedStatus接口的调用结果和更新数量
- 非成功状态的跳过日志

## 错误处理

- 外部接口调用失败时，不会影响采集任务的正常创建
- 会在日志中记录错误信息
- 如果无法获取is_new状态，则不会添加isnew参数
- updateProbedStatus调用失败时，不会影响用例执行结果上报的主流程
- 所有外部接口调用都有完整的异常处理和日志记录

## 注意事项

1. 确保外部接口服务正常运行
2. 外部接口的host配置需要正确
3. 用例编号中包含 `_ios_` 关键字才会被识别为iOS应用
4. 只有APP字段不为空的用例才会参与检查
5. 系统会为每个APP查询iOS和非iOS两种版本，然后根据用例的is_ios状态匹配结果
6. 如果查询不到匹配的结果，会使用默认值isnew=0
7. **只有用例采集状态为成功（SUCCESS或COMPLETED）时，才会调用updateProbedStatus接口**
8. 失败的用例不会触发探测状态更新，避免误报
