# 用例参数下发执行机功能改进说明

## 功能概述

本次改进实现了**根据配置的用例参数下发执行机**的功能，确保用例下发执行机时能够正确传递用例级别的自定义参数。

## 问题分析

### 原有问题

1. **参数传递不完整**：
   - 只传递了任务级别的自定义参数（`CollectTask.customParams`）
   - 没有传递策略级别的自定义参数（`CollectStrategy.customParams`）
   - 没有传递用例级别的自定义参数（`CollectStrategy.testCaseCustomParams`）

2. **用例参数未按用例过滤**：
   - 执行机接收到的是全局参数，无法区分不同用例的参数
   - 没有根据具体执行的用例ID获取对应的参数配置

3. **参数优先级不明确**：
   - 任务参数、策略参数、用例参数的优先级关系不清晰
   - 参数覆盖机制不完善

## 解决方案

### 1. 参数获取层次化

实现了三层参数获取机制：

```java
// 1. 任务级别参数（最高优先级）
String taskCustomParams = collectTask.getCustomParams();

// 2. 策略级别参数（中等优先级）
String strategyCustomParams = collectStrategy.getCustomParams();

// 3. 用例级别参数（最低优先级，但最具体）
String testCaseCustomParams = collectStrategy.getTestCaseCustomParams();
```

### 2. 用例参数过滤

新增 `filterTestCaseCustomParams()` 方法：

```java
/**
 * 过滤用例自定义参数，只返回当前执行实例中用例的参数
 */
private String filterTestCaseCustomParams(String testCaseCustomParams, List<TestCaseExecutionInstance> instances) {
    // 解析用例自定义参数JSON
    Map<String, List<Map<String, String>>> allTestCaseParams = objectMapper.readValue(
        testCaseCustomParams, 
        new TypeReference<Map<String, List<Map<String, String>>>>() {}
    );
    
    // 获取当前执行实例中的用例ID
    Set<Long> currentTestCaseIds = instances.stream()
        .map(TestCaseExecutionInstance::getTestCaseId)
        .collect(Collectors.toSet());
    
    // 过滤出当前用例的参数
    Map<String, List<Map<String, String>>> filteredParams = new HashMap<>();
    for (Map.Entry<String, List<Map<String, String>>> entry : allTestCaseParams.entrySet()) {
        Long testCaseId = Long.valueOf(entry.getKey());
        if (currentTestCaseIds.contains(testCaseId)) {
            filteredParams.put(entry.getKey(), entry.getValue());
        }
    }
    
    return objectMapper.writeValueAsString(filteredParams);
}
```

### 3. 参数合并机制

实现了参数合并逻辑：

```java
StringBuilder combinedParams = new StringBuilder();

// 1. 添加任务参数
if (taskCustomParams != null && !taskCustomParams.trim().isEmpty()) {
    combinedParams.append(taskCustomParams);
}

// 2. 添加策略参数
if (strategyCustomParams != null && !strategyCustomParams.trim().isEmpty()) {
    if (combinedParams.length() > 0) {
        combinedParams.append(",");
    }
    combinedParams.append(strategyCustomParams);
}

// 3. 添加用例参数
if (filteredTestCaseParams != null && !filteredTestCaseParams.trim().isEmpty()) {
    if (combinedParams.length() > 0) {
        combinedParams.append(",");
    }
    combinedParams.append(filteredTestCaseParams);
}
```

## 参数传递流程

### 1. 数据流向

```
采集策略配置
    ↓
CollectStrategy.testCaseCustomParams (JSON格式)
    ↓
CollectTaskProcessServiceImpl.getTaskCustomParams()
    ↓
参数过滤和合并
    ↓
TestCaseExecutionRequest.taskCustomParams
    ↓
执行机接收参数
    ↓

    ↓

```

## 功能特性

### 1. 参数优先级

- **任务参数** > **策略参数** > **用例参数**
- 高优先级参数会覆盖低优先级参数
- 用例参数最具体，适用于特定用例的配置

### 2. 参数过滤

- 只传递当前执行实例中涉及的用例参数
- 避免传递无关用例的参数，提高效率
- 支持多用例并发执行时的参数隔离

### 3. 错误处理

- JSON解析失败时记录警告日志，不影响其他参数传递
- 参数格式错误时跳过该参数，继续处理其他参数
- 提供详细的日志记录，便于问题排查

## 使用场景

### 场景1：不同用例需要不同超时时间

```json
// 策略配置
{
  "1": [{"key": "timeout", "value": "60"}],  // 视频通话用例
  "2": [{"key": "timeout", "value": "120"}], // 文件传输用例
  "3": [{"key": "timeout", "value": "30"}]   // 消息发送用例
}
```

### 场景2：特定用例需要特殊配置

```json
// 策略配置
{
  "1": [
    {"key": "concurrentUsers", "value": "100"},
    {"key": "duration", "value": "300"}
  ],  // 性能压测用例
  "2": []  // 功能测试用例（无特殊参数）
}
```

### 场景3：环境相关参数

```json
// 策略配置
{
  "1": [
    {"key": "dbHost", "value": "192.168.1.100"},
    {"key": "dbPort", "value": "3306"}
  ],  // 数据库连接测试用例
  "2": [
    {"key": "apiEndpoint", "value": "http://test.api.com"},
    {"key": "apiVersion", "value": "v2"}
  ]   // API接口测试用例
}
```

## 注意事项

### 1. 参数格式要求

- 用例自定义参数必须是有效的JSON格式
- 参数键值对必须包含 `key` 和 `value` 字段
- 用例ID必须与数据库中的用例ID一致

### 2. 参数合并规则

- 参数按优先级合并，高优先级覆盖低优先级
- 相同key的参数会被覆盖
- 空参数会被自动过滤

### 3. 错误处理

- JSON解析失败不会影响其他参数传递
- 参数格式错误会记录警告日志
- 建议在配置参数前验证JSON格式
